변경 이력
v131 v130에서 찾은 "갱신 체인 중 하나 실패 시 나머지가 통째로 안 도는" 문제를 전수점검 — 같은 패턴이 남아있는 곳이 없는지 코드 전체를 다시 훑음. 가장 심각했던 곳은 전표 수정 저장(`saveOrderEdit`, 공유·비공유 둘 다) — 여기서 실패하면 수정 모달이 안 닫히고 완료 토스트도 안 뜰 수 있었음. 그 외 납품 등록 확정(명세서 자동 오픈이 아예 무산될 수 있었음), 전표 삭제, 거래처 등록/수정/삭제(스와이프 삭제 포함), 오프라인 공유 납품 업로드 알림까지 총 5곳 추가로 동일하게 수정 — 전부 지금 보고 있는 화면부터 최우선으로, 각각 독립 실행되도록 통일. Firebase 쓰기 오류 처리도 전체적으로 점검했고 이상 없음 확인.
v130 완납→미수 복귀(및 수금 취소·수정)를 해도 열려있는 명세표 화면에 반영이 안 될 수 있던 문제 수정 — 결제/미수복귀 처리 후 여러 화면(전표 목록·대시보드·뱃지·명세표 등)을 한 줄로 쭉 이어서 갱신하던 코드가 13곳 있었는데, 그중 어느 하나라도 도중에 예외를 던지면 그 뒤에 나오는 갱신 호출은 아예 실행되지 않는 구조였음 — 특히 명세표 갱신이 뒤쪽에 있던 곳들은 데이터는 이미 바뀌었는데 화면만 안 바뀐 것처럼 보일 수 있었음. 지금 사용자가 보고 있는 화면(명세표)부터 최우선으로, 그리고 각 갱신을 서로 독립적으로 실행하도록 고쳐서 하나가 실패해도 나머지는 정상적으로 갱신됨.
v129 업데이트해도 일부 파일만 옛날 내용으로 남아있을 수 있던 근본 원인 수정 — 서비스워커가 새 파일을 받아올 때(설치 시 전체 캐싱, 백그라운드 재검증 둘 다) 브라우저 자체의 HTTP 캐시를 그대로 거쳐가고 있었음. 그래서 서비스워커 자체는 정상적으로 새 버전으로 교체돼도, 그 안에 채워 넣는 파일 내용이 브라우저 HTTP 캐시에 남아있던 옛 바이트일 수 있었음(서비스워커 입장에선 구분할 방법이 없음) — 예: index.html의 버전 표시는 v128로 바뀌었는데 js/app.js 같은 실제 동작 파일은 여전히 그 이전 코드로 실행되는 식. 모든 자산 fetch에 cache:'reload'를 명시해서 브라우저 HTTP 캐시를 완전히 건너뛰고 항상 네트워크에서 진짜 최신 파일을 받아오도록 수정.
v128 납품 메모 중요도(색상) 표시 기능 추가 — 메모 저장 시 🔵낮음 / 🟠보통(기본) / 🔴높음 3단계 중요도를 선택할 수 있고, 내역 카드의 메모 버튼·메모 본문, 전표 상세, 정산 화면, 메모 모아보기(요약 카드·상세 목록)까지 선택한 색으로 표시됨. 공유 내역(대납)에도 동일하게 적용. 기존 메모는 모두 보통(주황)으로 표시됨. 메모를 삭제하면 중요도도 함께 기본값으로 초기화되어, 다음에 새로 메모를 달 때 이전 중요도가 남아있지 않음.
v127 "새 버전이 준비되었습니다" 배너가 실제로 새 버전이 없어도 계속 재표시되던 문제 수정 — v126에서 배너 기능을 넣으며 함께 활용한 SW_UPDATED 메시지가, 실제 내용 변경 여부와 무관하게 백그라운드 재검증(stale-while-revalidate)이 성공할 때마다(즉 앱을 열 때마다) 무조건 발송되고 있었음. 그 결과 새로고침으로 최신 버전을 이미 적용한 직후에도 배너가 다시 나타나 사실상 저절로 사라지지 않는 것처럼 보였음 — 서비스워커가 캐시된 파일과 새로 받아온 파일 내용을 직접 비교해서, 실제로 달라졌을 때만 알림을 보내도록 수정. 이제 진짜 새 배포가 있을 때만 배너가 뜨고, 새로고침으로 적용하고 나면 다시 뜨지 않음.
v126 배포 후 새 버전이 자동으로 반영되지 않던 문제 개선 — 깃허브·Vercel 배포는 정상이어도, PWA 자체 서비스워커 캐시(stale-while-revalidate) 때문에 사용자가 수동으로 캐시를 지우거나 앱을 재설치하기 전까진 계속 옛 버전이 보였음. ① 서비스워커 등록 시 updateViaCache:'none' 적용 — 브라우저가 sw.js 파일 자체를 HTTP 캐시에서 재사용해 새 배포 감지가 늦어지는 것을 방지 ② 새 서비스워커가 설치·활성화되는 것을 감지(updatefound/statechange, 최초 설치는 제외)하면 "🆕 새 버전이 준비되었습니다 · 새로고침" 안내 배너를 자동으로 띄우도록 추가 — 기존에 이미 있었지만 아무도 듣고 있지 않던 SW_UPDATED 메시지도 보조 트리거로 함께 활용 ③ 화면(탭)으로 돌아올 때마다 능동적으로 업데이트 여부를 확인하도록 추가 — 기존엔 브라우저 기본 주기(최대 24시간)에만 의존하고 있었음.
v125 거래처별 "최근 품목" 자동완성이 10개까지만 보이던 제한을 30개로 상향 — 품목 종류가 10개를 넘는 거래처는 11번째 이후 품목이 납품 등록 화면의 "⚡ 이 거래처 최근 품목" 칩 목록뿐 아니라 "이 거래처 최근 단가" 자동완성에서도 영구히 안 나타났음. 원인은 두 겹: 거래처별 최근 품목을 모아두는 캐시 자체가 거래처당 10개로 잘려있었고(1차 원인), 화면에 뿌려주는 함수의 기본 표시 개수도 별도로 10개였음(2차 원인) — 둘 다 30개로 올려 실제로 더 보이도록 수정. 칩 영역은 줄바꿈(flex-wrap) 레이아웃이라 개수가 늘어도 잘리지 않고 자연스럽게 세로로 늘어남.
v124 할인 완납 전표가 거래처별 정산 화면에 미수로 남아 보이던 버그 수정 — 할인 완납 처리된 전표는 실청구액(청구액-할인액)이 실제 청구 기준이 되어야 하는데, 다음 3곳이 raw 청구액(discount 반영 전)을 그대로 합산하고 있어서 할인액만큼이 계속 미수로 표시됐음: ① 월별 정산 화면의 거래처별 테이블(💡 거래처 클릭 시 상세 명세서 목록) ② 정산서 Excel의 분기별 내보내기 매출·미수 컬럼 ③ 거래처 카드의 "총 매출" 표시. 세 곳 모두 이미 존재하던 실청구액 계산 헬퍼(할인 완납 시 total-discount)를 쓰도록 수정. 일별·분기별 인앱 화면, 거래명세표, 월별/일별 Excel 내보내기는 이미 정확히 계산되고 있어 변경 없음.
v123 거래명세표 월 이동 기능 추가 — 거래명세표 헤더에 ‹ › 버튼으로 이전/다음 달 이동, 제목 탭으로 월 선택(달력 picker) 가능. 자동 갱신 지점(전표 삭제·수정 시 명세표 즉시 새로고침)에서 원래 참조하던 data-client-name·data-month 속성이 실제로는 어디에도 없어 동작하지 않던 기존 사각지대도 이번에 헤더 요소에 속성을 부여하며 함께 정상화됨. 추가로 "💳 수금 이력" 섹션을 기본 접힘 상태로 변경 — 헤더 탭으로 펼침/접힘 토글, 건수·합계 요약은 접힌 상태에서도 표시. — 전체 검수 중 flushSync()에 남아있던 견고성 허점도 보강: v122에서 CRM 병합을 위해 await를 추가하면서, 가드 선점 이후~업로드 시작 전 구간(해시 계산·CRM 조회·병합)에서 예기치 못한 예외가 나면 어떤 catch도 없어 "저장 진행 중" 가드가 영구히 풀리지 않아 이후 모든 flushSync가 계속 스킵될 수 있는 위험이 있었음 — try/catch로 감싸 어떤 경로로 실패하든 가드가 반드시 해제되도록 보강.
v122 공유 거래처 대납 재고 미차감 버그 수정 — 공유 거래처에 대납 납품 시 실제로는 내 재고에서 출고되는 거래인데도, 기존 코드가 이를 타인거래(👤)와 동일하게 취급해 자동 재고 차감을 스킵하고 있었음. ① 대납 등록 시 정상적으로 내 재고 차감(반품/회수는 입고) 처리하도록 수정 ② 대납 전표 삭제 시 재고 복구 확인창이 뜨도록 추가(기존엔 삭제해도 재고 미반영) ③ 대납 전표 수정 시 수정 전후 수량 차이만큼 재고를 보정하도록 추가(기존엔 수정해도 재고 미반영) ④ 재고 탭 🔄 재계산 버튼이 delegatedBy(대납) 전표를 걸러내지 않아 상대방이 대납한 물량이 내 재고에서 또 차감되던 버그 수정. 타인거래(👤) 토글은 기존대로 재고 차감 없음 유지. — 추가로, 위 수정 과정에서 더 심각한 동기화 버그를 발견해 함께 수정: flushSync·debouncedSync(full)·재접속 시 "로컬이 최신" 업로드 3곳이 모두 서버의 orders 노드 전체를 내 로컬 스냅샷으로 통째로 교체하고 있었음. 이 때문에 상대방이 대납으로 내 워크스페이스에 직접 써넣은 전표가, 아직 내 로컬에 실시간 반영되기 전 타이밍에 내가 무관한 다른 저장(납품 등록·수금 등)을 하거나 오프라인 후 재접속하면 통째로 삭제되는 문제가 있었음(재고는 대납한 사람 자신의 워크스페이스에 저장되므로 안 사라지고, 전표만 사라져 보였음). 3곳 모두 orders/{id} 경로별 병합 방식으로 변경해 언급되지 않은 항목(=아직 못 받은 대납 전표)은 건드리지 않도록 수정. 이후 CRM 연동 점검 중 flushSync만 CRM 결제 필드 보존 원칙(debouncedSync는 지키던 것)을 지키지 않던 것도 함께 발견해 서버 값 병합 로직 추가, 그 과정에서 생긴 재진입 가드 허점도 보강.

v109 화면 확대(핀치줌) 간헐적 새어나가는 문제 수정 — CSS touch-action을 전역(*)과 버튼·입력창·카드류에서 manipulation으로 설정해왔는데, manipulation은 명세상 더블탭줌만 막고 핀치줌(연속 확대)은 허용하는 값이라 JS의 핀치줌 차단 로직과 어긋나 있었음(브라우저가 네이티브 줌 제스처를 먼저 처리하면 간헐적으로 확대가 새어나감). touch-action을 pan-x pan-y로 변경해 핀치줌·더블탭줌 모두 CSS 레벨에서 1차 차단하고, 기존 JS 차단은 보조 안전장치로 유지. 탭 반응속도(300ms 지연 제거) 효과는 동일하게 유지됨.
